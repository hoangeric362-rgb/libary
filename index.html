1. GIỚI THIỆU ĐỀ TÀI

1.1. Bối cảnh & Lý do chọn đề tài

  • Thời đại số hóa: Yêu cầu quản lý lượng lớn tài nguyên (sách) một cách hiệu quả.
  • Vấn đề của cấu trúc tuyến tính: Với List hoặc Array, các thao tác tìm kiếm, cập nhật có độ phức tạp O(N).
  • Thách thức: Khi N lớn (hàng trăm ngàn đầu sách), O(N) gây lãng phí tài nguyên và thời gian chờ đợi.
  • Giải pháp: Sử dụng cấu trúc phi tuyến tính Hash Table (Dictionary trong C#).
  • Ưu điểm: Hash Table cung cấp độ phức tạp trung bình O(1) cho các thao tác Thêm, Xóa, Tìm kiếm, Cập nhật.

Kết luận: Nhóm chọn đề tài này để triển khai và chứng minh tính ưu việt về hiệu suất của Hash Table trong bài toán quản lý thực tế.

1. GIỚI THIỆU ĐỀ TÀI

1.2. Mục tiêu

  • Xây dựng một mô hình quản lý dữ liệu sách (Book) tối ưu về tốc độ.
  • Chứng minh sự khác biệt về hiệu suất (tốc độ tải, truy xuất) giữa Hash Table và cấu trúc tuyến tính qua thử nghiệm dữ liệu lớn.
  • Thiết kế giao diện Console đơn giản, cho phép người dùng thực hiện các chức năng CRUD cơ bản (Create, Read, Update, Delete).
  • Đánh giá khả năng mở rộng của Hash Table khi xử lý dữ liệu lớn (trên 10.000 bản ghi).

1. GIỚI THIỆU ĐỀ TÀI

1.3. Phạm vi, giới hạn

  • Phạm vi: Quản lý thông tin sách (BookID, Title, Author...) và các thao tác quản trị.
  • Giới hạn:
    • Chỉ sử dụng Console App (không có GUI).
    • Chỉ tập trung vào Hash Table / Dictionary.
    • Dữ liệu lưu trữ cục bộ (file JSON), không dùng DBMS.

1.4. Phương pháp thực hiện

  • Nghiên cứu lý thuyết: Phân tích độ phức tạp thời gian O(T).
    • Hash Table: O(1)
    • List / Array: O(N)
  • Lập trình: Sử dụng C# (.NET Core) và thư viện Newtonsoft.Json.
  • Thử nghiệm: Tạo 10.000+ sách giả để đo lường thời gian thực thi.

2. PHÂN TÍCH YÊU CẦU

2.1 & 2.2. Khảo sát và Thu thập

  • Hệ thống mô phỏng kho dữ liệu thư viện cơ bản.
  • Yêu cầu nghiệp vụ cốt lõi: Mỗi cuốn sách cần được truy cập nhanh chóng bằng một mã định danh duy nhất (BookID).
  • Yêu cầu bắt buộc: Tốc độ truy xuất dữ liệu (tìm theo ID) phải gần như tức thì.

2.3. Mô hình phân tích dữ liệu

  • Thực thể chính: Book (Sách).
  • Khóa chính: BookID (string) → Sử dụng làm Key trong Hash Table.
  • Cấu trúc dữ liệu trong bộ nhớ:
    Dictionary<string, Book> libraryData;
    • Key: BookID (string)
    • Value: Book (object)

2. PHÂN TÍCH YÊU CẦU

2.4. Yêu cầu chức năng (FNs)

Mã Tên Chức năng Mô tả chi tiết
FN1 Thêm sách mới Cho phép nhập thông tin sách và tự động tạo BookID.
FN2 Cập nhật sách Cập nhật thông tin sách dựa trên BookID (Yêu cầu O(1)).
FN3 Xóa sách Xóa sách khỏi hệ thống dựa trên BookID.
FN4 Tìm kiếm sách Tìm kiếm theo BookID (Yêu cầu O(1)) hoặc theo Tên/Tác giả (Chấp nhận O(N)).
FN5 Hiển thị sách Hiển thị toàn bộ danh sách sách (Dictionary.Values).
FN6 Quản lý Dữ liệu Lớn Chức năng tạo dữ liệu giả tự động để thử nghiệm hiệu năng.

2. PHÂN TÍCH YÊU CẦU

2.5. Yêu cầu phi chức năng (NFRs)

Mã Tên Yêu cầu Chi tiết
NFR1 Hiệu suất Thao tác Tìm kiếm, Thêm, Xóa theo ID phải đạt độ phức tạp O(1).
NFR2 Độ tin cậy Dữ liệu phải được lưu trữ vĩnh viễn (JSON) và tải lại chính xác khi khởi động.
NFR3 Khả năng mở rộng Hệ thống phải xử lý hiệu quả khi số lượng sách vượt quá 10.000 bản ghi.

3. THIẾT KẾ HỆ THỐNG

3.1. Kiến trúc hệ thống (3 Lớp)

  • Lớp Giao diện (Program):

  • Lớp Logic (LibraryManager):

  • Lớp Dữ liệu (Book):
    • Định nghĩa cấu trúc dữ liệu của đối tượng Sách.

3.2. Xây dựng dữ liệu khởi tạo

3. THIẾT KẾ HỆ THỐNG

3.3. Thiết kế chức năng (Logic & Độ phức tạp)

  • Tải dữ liệu (LoadAllBooks):
    • Đọc file library_data.json → Deserialize vào Dictionary.
    • Độ phức tạp: O(F) (F là kích thước file), thực hiện một lần khi khởi động.
  • Thêm / Xóa / Cập nhật (theo ID):
    • Sử dụng phương thức tích hợp của Dictionary:
      currentLibrary.Add(key, value)
      currentLibrary.Remove(key)
      currentLibrary.TryGetValue(key, out Book)
    • Độ phức tạp lý thuyết: O(1).
  • Tìm kiếm (SearchBooks theo Tên/Tác giả):
    • Phải duyệt qua Dictionary.Values.
    • Độ phức tạp: O(N) (nhưng nhanh vì duyệt trong bộ nhớ).
  • Lưu dữ liệu (SaveAllBooks):
    • Serialize Dictionary → Ghi đè toàn bộ file JSON.
    • Độ phức tạp: O(N) (Đây là một "nút thắt cổ chai").

4. TRIỂN KHAI VÀ KẾT QUẢ

4.1. Công cụ, môi trường phát triển

  • Ngôn ngữ lập trình: C# (.NET Core 8)
  • IDE: Visual Studio / VS Code
  • Thư viện ngoài: Newtonsoft.Json (Quản lý Serialize/Deserialize)
  • Cấu trúc dữ liệu chính: Dictionary<string, Book> (Triển khai của Hash Table trong C#)

4.3. Các chức năng đã cài đặt

  • Tất cả các chức năng từ FN1 đến FN6 đã được cài đặt thành công.
  • Chức năng Tạo Sách Tự động (Generate Large Data) đã tạo thành công 10.000 bản ghi và lưu trữ ổn định.
  • Giao diện Console được tùy chỉnh (màu sắc, căn chỉnh) để cải thiện trải nghiệm.

4. TRIỂN KHAI VÀ KẾT QUẢ

4.4. Kết quả kiểm thử (Với 10.000 bản ghi)

Thao tác Cấu trúc dữ liệu Thời gian thực hiện (TB) Độ phức tạp (Lý thuyết)
Tải 10.000 sách Hash Table (1 file JSON) 50 - 150 ms O(F)
Tìm kiếm theo ID Hash Table (Dictionary) < 1 ms (Tức thì) O(1)
Thêm sách Hash Table (Dictionary) 150 - 300 ms O(1) + O(Save)
Tìm kiếm theo Tên Hash Table (Values) 5 - 10 ms O(N)

Ghi chú: Thao tác "Thêm sách" bị chậm chủ yếu do thời gian Lưu file JSON (O(Save)) chứ không phải do thao tác Add của Dictionary (vốn là O(1)).

5. ĐÁNH GIÁ VÀ KẾT LUẬN

5.1. Đánh giá kết quả đạt được

  • Hoàn thành mục tiêu: Ứng dụng hoạt động ổn định và đáp ứng các yêu cầu.
  • Chứng minh hiệu suất: Thử nghiệm đã chứng minh tính ưu việt của Hash Table với độ phức tạp O(1) cho các thao tác truy xuất chính (Tìm theo ID).
  • Tối ưu I/O: Việc dùng 1 file JSON duy nhất đã tối ưu hóa vượt trội thời gian tải ứng dụng khi khởi động.

5. ĐÁNH GIÁ VÀ KẾT LUẬN

5.2. Những khó khăn, hạn chế

  • Vấn đề I/O (Nút thắt cổ chai):
    • Việc lưu toàn bộ dữ liệu (SaveAllBooks) ra file JSON mỗi khi Thêm/Xóa/Sửa là một điểm nghẽn.
    • Nó làm giảm tốc độ thực thi của các thao tác này từ O(1) (lý thuyết) xuống O(Save) (thực tế).
  • Khả năng va chạm (Collision):
    • Về lý thuyết, Hash Table có rủi ro va chạm làm giảm hiệu suất.
    • Tuy nhiên, Dictionary của C# được tối ưu rất tốt nên không ảnh hưởng đáng kể.
  • Giao diện: Giao diện Console còn thô sơ.

5. ĐÁNH GIÁ VÀ KẾT LUẬN

5.3. Hướng phát triển

  • Cải thiện I/O:
    • Sử dụng hệ thống Cơ sở dữ liệu (như SQLite hoặc Firebase).
    • Lợi ích: Chỉ lưu các thay đổi (Delta changes) thay vì ghi đè toàn bộ file, giúp thao tác Thêm/Xóa/Sửa đạt O(1) thực sự.
  • Mở rộng Cấu trúc:
    • Áp dụng B-Tree (hoặc cấu trúc cây) cho phép tìm kiếm theo phạm vi (Range Search).
    • Ví dụ: Tìm sách xuất bản trong khoảng năm 2010 - 2020.
  • Giao diện:
    • Chuyển sang Web/Desktop hoặc thư viện Console nâng cao (Spectre.Console).

5. ĐÁNH GIÁ VÀ KẾT LUẬN

5.4. Kết luận

Đồ án đã thành công trong việc áp dụng cấu trúc dữ liệu Hash Table để giải quyết bài toán hiệu suất trong quản lý thư viện.

Kết quả từ các thử nghiệm trên 10.000 bản ghi đã chứng minh rõ ràng lợi ích về tốc độ truy xuất (O(1)).

Qua đó, nhóm đã củng cố thêm kiến thức về tầm quan trọng của việc lựa chọn cấu trúc dữ liệu phù hợp với yêu cầu bài toán.
